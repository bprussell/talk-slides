An open standard for packaging instructions, scripts, and resources — and only loading them when the agent actually needs them.

04 / Skills
Skills · Anatomy

A skill is a folder.

Frontmatter, a body of instructions, and whatever scripts or references the agent might need to do the job.

SKILL.md
---
name: timesheet-recap
description: Recap what I worked
  on this pay period by searching
  Slack & Outlook, then draft a
  timesheet entry I can review.
---

# Timesheet Recap

When the user asks what they
worked on, search Slack DMs
and Outlook calendar for the
active pay period...
Folder
timesheet-recap/
├── SKILL.md          # the prompt
├── scripts/
│   └── pay_period.py # helper
└── references/
    └── eg_categories.md

Only the description ships in the agent's base context. The body and files load when the skill matches — same format in Claude Code, the API, and Cowork.

05 / Skills
Skills · Why it matters

Progressive disclosure for agents.

Skills solve the same problem libraries solved for code: you don't load every function into memory just in case.

  • Tokens stay cheap — instructions only enter context when relevant.
  • You can ship dozens of skills without poisoning the context window.
  • The body can include scripts the agent runs, not just text it reads.
  • It's a portable standard — the same SKILL.md folder works across Claude Code, Anthropic API, plus other tools picking up the format.

The catch (Thoughtworks calls this out)

  • Unreviewed third-party skills = supply chain risk. A SKILL.md can ship arbitrary scripts. Review what you install.

What it does

Built in Claude Cowork. Looks at my Slack DMs and Outlook calendar for the current pay period, summarizes what I worked on, drafts a timesheet entry to review.

Why a skill

No new server. No OAuth flow. Just instructions + a small script. Slack & Outlook are already reachable via MCP — the skill composes what's there.

Watch for

Status

08 / Marketplace
Marketplace · Mechanism

Skills, commands, prompts — versioned.

A Claude Code plugin is a bundle. A marketplace is a Git repo of bundles. Anyone can host one — internally, publicly, or both.

marketplace.json
{
  "name": "eg-claude-plugins",
  "owner": "e-gineering",
  "plugins": [
    {
      "name": "timesheet",
      "source": "./plugins/timesheet",
      "version": "0.2.0"
    }
  ]
}
  • /plugin marketplace add <repo> — point Claude Code at a Git URL.
  • /plugin install timesheet — sync the bundle locally.
  • Updates are pulls, not copy-paste from Confluence.
  • A plugin can contain skills and commands and hooks and an MCP server.

10 / Marketplace
Marketplace · The other path
Path B · Built into the product

Cowork on a team plan: publish to your org directory.

Same SKILL.md format. Different distribution channel — managed by Anthropic, scoped to your org, no Git repo required.

How it works

  • Build the skill in Cowork (or upload an existing SKILL.md folder).
  • Share with a colleague, or publish to the org directory.
  • Anyone in the org finds it and installs in one click.
  • You push an update → everyone gets it automatically.

Why it complements the radar entry

  • Git marketplace — open, portable, public-or-private, works for Claude Code.
  • Cowork org directory — managed, internal-only, propagates updates without Git plumbing.
  • Same SKILL.md. Pick the channel that fits the audience.

Requires Team or Enterprise plan. Org owner enables Code execution + File creation in capabilities first.

14 / Caution
Caution · When to reach for MCP

The abstraction tax is real.

Every protocol layer between agent and API costs fidelity. Complex APIs compound the loss.

Skip MCP when

  • A CLI with --help, JSON output, and clear exit codes already exists.
  • A skill with a small script will do the job.
  • Only one tenant, one team, one machine.

MCP earns its place when

  • You need OAuth boundaries (multi-user, governed access).
  • You need structured tool contracts across many clients.
  • You need multi-tenant governance from one server.

16 / MCP Apps
MCP Apps · How it works

A new resource scheme: ui://

Tool returns a UI template + data → host renders it in a sandboxed iframe. Text fallback if the client doesn't support UI.

tools/call response (sketch)
{
  "content": [
    { "type": "text",
      "text": "Vacation: 12.5 days left" },
    { "type": "resource",
      "resource": {
        "uri": "ui://vacation-report",
        "mimeType": "text/html"
      } }
  ]
}
  • Bidirectional — the model can observe user actions inside the iframe.
  • Sandboxed — host controls what the iframe can touch.
  • Graceful — text fallback keeps non-UI clients working.
  • Standard — first official extension. Anthropic + OpenAI + community.
17 / MCP Apps
MCP Apps · Where it works

Already shipped in the clients that matter.

Build the UI once, render in any compliant host. Same MCP server, four front-ends.

Claude ChatGPT VS Code Goose

Compare to AG-UI Protocol

  • AG-UI is a separate library bolted on. MCP Apps lives inside the MCP server — one place to ship behavior + UI.
18 / MCP Apps
Demo · 02

EG MCP — ChatGPT Apps today, MCP Apps tomorrow.

Today

Our internal MCP server renders the handbook, perk wallet, timesheet, and vacation report through the ChatGPT Apps SDK. Works great — in ChatGPT.

Constraint

The UI is bound to one host. Claude Code can't render it. VS Code can't render it. Anything not-ChatGPT just sees the raw JSON.

Tomorrow

Same server, MCP Apps resources. Handbook viewer, perk wallet, timesheet — portable. This talk is partly the motivation to start that migration.

Watch for

Live: the perk wallet via ChatGPT Apps today. Sketch: what the ui://perk-wallet resource would look like.
